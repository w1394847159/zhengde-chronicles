## 1. 项目概述

### 1.1 游戏定位

- **名称**：正德风云录（英文名：ZhengdeChronicles）
- **类型**：AI Native 历史策略模拟游戏
- **平台**：Android（Kotlin + Jetpack Compose）
- **题材**：明武宗朱厚照正德朝（1505–1521），后续可扩展多朝代
- **核心玩法**：玩家扮演朱厚照，以自然语言下达诏书，LLM 推演影响世界数值状态，驱动历史走向
- **色调风格**：UI 以暗金/朱红/墨色为主，仿明清线装书质感

### 1.2 核心体验

正德朝与崇祯朝的本质区别：

| 维度 | 崇祯朝（亡国压抑） | 正德朝（疯批戏剧） |
|------|------------------|------------------|
| 玩家情绪 | 焦虑、绝望、负重前行 | 狂浪、刺激、作死与救火 |
| 核心矛盾 | 救亡图存 | 玩 vs 治 的拉扯 |
| 历史结局 | 亡国（几乎必然） | 多种走向（可兴可亡可乱） |
| 节奏 | 步步紧逼 | 间歇性爆炸+日常摸鱼 |

### 1.3 设计原则

1. **AI Native** — 剥离 LLM 游戏不可玩，LLM 不是辅助而是核心引擎
2. **数值驱动** — LLM 输出必须转化为真实可查的数值变化，而非纯文案
3. **沉浸优先** — 仿明白话叙事，明清小说质感
4. **可打断** — 单次诏书推演控制在 2 分钟内完成
5. **零数据收集** — 纯本地 App，用户自配 API Key

### 3.2 数据流（一次完整回合）

```
① 玩家输入诏书文本
    ↓
② EdictViewModel 接收 → 存入本地临时状态
    ↓
③ WorldEngine 触发推演回合
    ↓
④ PromptBuilder 组装 Prompt：
    - 当前世界状态快照（数值）
    - 历史关键事件摘要（MemorySystem 压缩）
    - 玩家诏书原文
    - 角色设定（朱厚照性格/当前环境）
    ↓
⑤ LLM Client → OkHttp → 用户配置的 API
    ↓
⑥ LLM 返回结构化 JSON（推演结果）
    ↓
⑦ EffectSystem 解析 JSON → 校验合理性 → 执行数值变更
    ↓
⑧ StateManager 生成新状态快照 → 存入 Room
    ↓
⑨ EventSystem 检测阈值 → 生成新事件/奏折
    ↓
⑩ MemorySystem 压缩本轮摘要 → 追加到长期记忆
    ↓
⑪ ViewModel 收集新状态 → UI 更新
```

## 4. 核心引擎设计

### 4.1 WorldEngine（世界引擎主循环）

这是游戏的心脏，负责调度一切。

```kotlin
class WorldEngine(
    private val stateManager: StateManager,
    private val effectSystem: EffectSystem,
    private val eventSystem: EventSystem,
    private val npcManager: NpcManager,
    private val memorySystem: MemorySystem,
    private val narrativeSystem: NarrativeSystem,
    private val llmClient: LlmClient,
    private val tokenTracker: TokenTracker
) {
    // 引擎状态
    sealed class State {
        data object Idle : State()
        data object Processing : State()
        data class AwaitingEdict(val turn: Int) : State()
        data class Error(val reason: String) : State()
    }
    
    // 执行一个完整回合
    suspend fun executeTurn(edict: PlayerEdict): TurnResult {
        // 1. 构建 Prompt
        val prompt = buildEdictPrompt(edict)
        
        // 2. 调用 LLM
        val rawResponse = llmClient.generate(prompt)
        tokenTracker.record(prompt, rawResponse)
        
        // 3. 解析 LLM 输出（结构化 JSON）
        val effect = effectSystem.parse(rawResponse)
        
        // 4. 合理性校验（硬约束）
        effectSystem.validate(effect, stateManager.currentState)
        
        // 5. 执行数值变更
        val newState = stateManager.apply(effect)
        
        // 6. 事件检测
        val newEvents = eventSystem.detect(newState, previousState)
        
        // 7. 记忆处理
        memorySystem.compress(edict, effect, newState)
        
        // 8. NPC 状态更新
        npcManager.tick(newState, effect)
        
        // 9. 返回结果
        return TurnResult(
            narrative = narrativeSystem.render(effect, newState),
            newState = newState,
            events = newEvents,
            tokenCost = tokenTracker.lastCost
        )
    }
}
```

**关键设计决策**：
- **同步阻塞式回合**：每回合必须等待 LLM 返回，不搞异步并行推理（状态一致性优先）
- **超时处理**：LLM 调用设 120 秒超时，超时后允许玩家"重推"或"跳过"
- **错误容忍**：LLM 返回格式错误时，自动重试 1 次+降级为有限推演

### 4.2 StateManager（步进状态机）

核心思路跟崇祯模拟器一致——**每个回合一个不可变状态快照，增量更新**。

```kotlin
class StateManager(
    private val db: GameDao
) {
    // 当前快照
    private var _currentState: WorldState? = null
    val currentState: WorldState get() = _currentState!!
    
    // 历史快照索引（按回合号）
    private val snapshots: SortedMap<Int, WorldState> = TreeMap()
    
    fun initialize(startYear: Int = 1505): WorldState {
        val state = WorldState.createDefault(startYear)
        _currentState = state
        snapshots[0] = state
        db.saveSnapshot(state)
        return state
    }
    
    fun apply(effect: EdictEffect): WorldState {
        val prev = _currentState!!
        val newState = prev.copy(
            turn = prev.turn + 1,
            timestamp = System.currentTimeMillis(),
            // 各维度增量更新
            treasury = clamp(prev.treasury + effect.treasuryDelta, 0, MAX_TREASURY),
            granary = clamp(prev.granary + effect.granaryDelta, 0, MAX_GRANARY),
            military = clamp(prev.military + effect.militaryDelta, 0, MAX_MILITARY),
            // ... 其余 n 个字段类似
            changeLog = prev.changeLog + effect.toChangeEntry()
        )
        _currentState = newState
        snapshots[newState.turn] = newState
        db.saveSnapshot(newState)
        // 异步清理旧快照（保留最近 50 回合、每 10 回合一个检查点）
        db.cleanupSnapshots()
        return newState
    }
    
    fun rollbackTo(turn: Int): WorldState? {
        val snapshot = snapshots[turn] ?: db.loadSnapshot(turn)
        if (snapshot != null) {
            _currentState = snapshot
            return snapshot
        }
        return null
    }
}
```

**增量 vs 全量**：
- 每回合只保存增量变更日志 + 关键维度的绝对值快照
- 全量快照每 10 回合保存一次（Room 存储，大小可控）
- LLM 只接收当前快照 + 最近 3 轮变更摘要（控制 Token 消耗）

### 4.3 EffectSystem（效果解析与校验系统）

这是介于 LLM 和游戏状态之间的**安全阀**。

```
LLM 输出（结构化 JSON）→ EffectSystem → 可执行的数值变更
                        ↓
                  校验门（硬约束）
                  - 数值不能超过上限/下限
                  - 不能凭空产生资源（必须有来源描述）
                  - 不能瞬间改变派系关系超过阈值
                  - 诏书意图与效果必须合理匹配
```

```kotlin
data class EdictEffect(
    val summary: String,                 // 推演叙事摘要（展示给玩家）
    val treasuryDelta: Int,              // 国库变化
    val granaryDelta: Int,               // 粮储变化
    val militaryDelta: Int,              // 军力变化
    val popularSupport: Map<String, Int>, // 各省民心变化
    val factionRelation: Map<String, Int>,// 各派系好感变化
    val emperorAttributes: Map<String, Int>, // 皇帝个人属性变化
    val provinceStatus: Map<String, ProvinceDelta>, // 各省状态变化
    val specialFlags: List<String>,       // 特殊标记（触发事件用）
    val unexpectedEvents: List<String>    // 连锁反应事件描述
)
```

**LLM Prompt 中的 JSON 约束**（关键设计）：

在 Prompt 中明确要求 LLM 按以下 Schema 输出：

```json
{
  "effect": {
    "treasury_delta": -120,          // 此次减120万两
    "granary_delta": -50,
    "military_delta": 15,
    "popular_support": {
      "北直隶": -5,
      "江西": -2
    },
    "faction_relation": {
      "八虎": 10,
      "内阁": -15
    },
    "emperor_attributes": {
      "威望": 5,
      "玩心": -10,
      "精力": -5
    },
    "narrative_summary": "陛下下令裁撤刘瑾党羽，八虎震动。内阁联名上表称颂圣明，然刘瑾爪牙暗中活动，朝局暗流涌动。",
    "trigger_events": [],
    "unexpected_consequences": "江西传来消息，宁王府近日频繁宴请地方官，似有异动。"
  }
}
```

**校验规则**：
- 所有 delta 绝对值不能超过 `MAX_SINGLE_DELTA`（单次变化上限，防失控）
- 总资源变化必须有合理解释（LLM 必须在 narrative 中说明来源去向）
- 史实严重偏离时（如"崇祯造原子弹"这种，但正德朝可以更宽松），标记为"演义模式"

### 4.4 MemorySystem（记忆系统）

崇祯模拟器制作人明确说过：**RAG 效果不好**，因为会让模型只关注玩家提到的关键词，忽略全局。我们的方案：

```
┌──────────────────────────────────────────┐
│               MemorySystem                │
│                                           │
│  短期记忆（当前回合完整状态）               │
│    ↓                                      │
│  中期记忆（最近 5 回合变更摘要）            │
│    ↓                                      │
│  长期记忆（压缩后的关键事件链）             │
│    - 以时间线方式存储                      │
│    - 每 5 回合压缩一次                     │
│    - 按重要性排序保留 TOP 50 条            │
│    ↓                                      │
│  Prompt 构建时按需取用：                    │
│  - 当前全量状态快照 → 必送                  │
│  - 最近 3 回合变更摘要 → 必送              │
│  - 长期记忆（最近 20 条关键事件）→ 含       │
│  - 与当前诏书相关的历史事件 → RAG 检索辅助  │
└──────────────────────────────────────────┘
```

**核心创新：分层记忆注入**

不直接把所有历史塞进 Prompt，而是：

```
Level 0: {当前状态快照}  → 全量数值
Level 1: {最近3轮摘要}   → 玩家刚做了什么
Level 2: {关键历史节点}  → 由 EventSystem 标记的里程碑事件
Level 3: {关联记忆}     → 语义相似度匹配的历史回合（轻量 RAG）
```

**为什么轻量 RAG 可行**：
- 只在 Level 3 用 RAG，且匹配范围仅限于"同类事件"（如玩家提到"江西"，检索江西相关历史）
- 不替代全局感知，只做补充

### 4.5 EventSystem（事件系统）

负责检测世界状态的变化阈值，自动生成"奏折"和"突发情况"。

```kotlin
class EventSystem {
    // 硬阈值规则（可配置）
    private val thresholds = listOf(
        Threshold("民变", TriggerType.LESS_THAN, "popular_support.陕西", 20),
        Threshold("边患", TriggerType.GREATER_THAN, "border_threat", 70),
        Threshold("国库空虚", TriggerType.LESS_THAN, "treasury", 500),
        // ...
    )
    
    // 检测函数，返回触发的事件列表
    fun detect(newState: WorldState, oldState: WorldState): List<GameEvent> {
        val events = mutableListOf<GameEvent>()
        
        // 1. 阈值检测
        for (threshold in thresholds) {
            if (threshold.isTriggered(newState)) {
                events.add(threshold.toEvent(newState))
            }
        }
        
        // 2. 剧烈变化检测（一次 drop 超过 30% 触发特殊事件）
        if (newState.treasury < oldState.treasury * 0.7) {
            events.add(GameEvent.剧烈变动("国库骤减", newState.treasury - oldState.treasury))
        }
        
        // 3. LLM 推演中标记的 trigger_events 直接入队
        // （由 EffectSystem 返回的 trigger_events 字段）
        
        // 4. 派系关系临界值检测
        for ((faction, relation) in newState.factionRelations) {
            if (relation < 10) {
                events.add(GameEvent.派系危机(faction, relation))
            }
        }
        
        return events
    }
}
```

事件生成后进入**事件队列**，下回合玩家必然会收到相关奏折，必须回应后才能继续正常治理。

### 4.6 NarrativeSystem（叙事系统）

负责把推演结果渲染成明清白话小说风格的文案。

**英文机制**：
- LLM 在输出 effect 时已经包含 `narrative_summary` 字段
- NarrativeSystem 在此基础上做后处理：
  - 替换称谓（"朕"、"陛下"、"公公"等）
  - 添加拟声词/感叹（"好！"、"罢了罢了"）
  - 按明清话本风格分段（"却说..."、"且说..."、"正是..."）

**模板示例**：
```
却说陛下[诏书核心摘要]，圣旨一下，[派系/官员]闻之[反应]。
[数值变化叙事化描述——如"国库又去了百万两白银，户部尚书抱着账本直哭"]
[连锁反应——民间/边关/朝堂的新动向]
正是：[收尾诗/对句]
```

## 5. 数据模型

### 5.1 WorldState（世界状态根）

```kotlin
data class WorldState(
    val turn: Int = 0,

    val courtStability: Int = 60,    // 朝纲（0-100）
```

### 6.3 模型选型建议

| 模型 | 性价比 | 推荐场景 |
|------|--------|---------|

| **千问 Qwen-Max** | ⭐⭐⭐⭐ | 当 DeepSeek 波动时作为备选 |
| **GLM-5** | ⭐⭐⭐⭐ | 长上下文（千万级 Token）场景有优势 |

**成本估算**（基于崇祯模拟器实际数据）：
- 单次推演 ≈ 4-6K Token（输入）+ 0.8-1.2K Token（输出）

- 单回合成本 ≈ 0.003-0.006 元
- 一次完整游戏（100 回合）≈ 0.3-0.6 元
- 重度玩家（1000 回合）≈ 3-6 元

**注意**：以上是纯 LLM API 成本，如果玩家用 DeepSeek 官方 API，比崇祯模拟器（他们加了一层服务+利润）便宜很多。

### 7.2 关键数值互锁关系

```
国库(treasury) ←→ 税收(各省 taxRevenue) ←→ 民心(各省 popularSupport)
    ↑                    ↑                        ↑
    ├── 加税 → 国库↑  → 民心↓  → 治安↓ → 可能民变
    ├── 减税 → 国库↓  → 民心↑  → 治安↑ → 长期利好
    │
军力(military) ←→ 国库(treasury)
    ↑                    ↑
    ├── 扩军 → 军力↑  → 国库↓↓  ← 军饷开支
    ├── 裁军 → 军力↓  → 国库↑   ← 省军费
    
边患(borderThreat) ←→ 军力(military) ←→ 边防省份驻军
    ↑                                        ↑
    └── 边患高 → 需要强驻军 ← 驻军强的省份边患低

玩心(playfulness) ←→ 精力(energy) ←→ 朝纲(courtStability)
    ↑                    ↑                ↑
    ├── 豹房玩耍 → 玩心↑精力↓ → 朝纲↓ ← 文官不满
    ├── 勤政 → 玩心↓精力↓↓ → 朝纲↑
```

### 7.3 动态阈值系统

数值不只是静态展示，它在不同区间有不同"质变"效果：

```kotlin
enum class RiskLevel(val range: IntRange, val label: String) {
    SAFE(60..100, "安定"),
    WATCH(30..59, "堪忧"),
    DANGER(10..29, "危急"),
    CRITICAL(0..9, "崩坏")
}
```

例如民心陕西从 60 降到 20，不只是数字变化：
- 60-100：正常纳粮
- 30-59：抗税事件概率上升
- 10-29：**触发民变阈值**，EventSystem 生成"流民作乱"事件
- 0-9：自动生成"造反/攻城"事件，不下诏处理也会恶化

**阈值表（部分）**：

| 维度 | 警告线 | 危险线 | 崩坏线 |
|------|--------|--------|--------|
| 民心 | < 30 | < 15 | < 5 |
| 国库 | < 500 | < 100 | < 0（负债） |
| 朝纲 | < 40 | < 20 | < 10 |
| 边患 | > 70 | > 85 | > 95 |
| 威望 | < 30 | < 15 | < 5 |
| 玩心 | > 80 | > 90 | 极值（特殊剧情） |

## 8. 特色系统设计

### 8.1 🐆 豹房系统

正德朝最具辨识度的玩法。

**设计思路**：
- 豹房是一个**独立的场景页面**，玩家可以"去豹房"（消耗回合）
- 在豹房里可以做特殊的事（召八虎、赴宴、练兵、躲清静）
- 去豹房会产生朝纲/民心的负面效果，但会解锁特殊能力/情报

**推演逻辑**：

```
去豹房（消耗 1 回合）
  ├── 召八虎饮宴
  │     ├── 八虎好感 +10
  │     ├── 朝纲 -8
  │     ├── 玩心 +5
  │     └── 解锁：刘瑾密报（情报优势）
  │
  ├── 边将夜谈
  │     ├── 边军好感 +8
  │     ├── 内阁 -5（文官不满）
  │     ├── 武艺 +3
  │     └── 解锁：亲征选项
  │
  ├── 独自静养
  │     ├── 精力 +15
  │     ├── 玩心 -5
  │     └── 无事发生
  │
  └── 特殊：王阳明求见（触发条件）
        ├── 促膝长谈 → 政治智慧 +10
        ├── 朝纲 +5
        └── 解锁：心学辅助决策
```

### 8.2 ⚔️ 亲征系统

朱厚照一生最大的标签——真刀真枪上战场。

**触发条件**：
- 边患 > 70 时，自动弹出"御驾亲征"选项
- 也可主动下诏亲征

**推演流程**：

```
玩家下诏"亲征"
    ↓
LLM 推演战局：
  - 边军战力 vs 蒙古军力
  - 后勤补给（国库/粮储）
  - 后方稳定（朝纲/各省民心）
  - 皇帝个人武艺值影响胜率
    ↓
结果（多种可能）：
  ├── ✓ 大胜：威望暴涨 +20，边患降至 30 以下，军力 +30
  ├── △ 小胜：威望 +8，边患 -15
  ├── ▲ 失利：威望 -10，军力 -15，可能受伤
  ├── ✗ 大败：威望暴跌 -25，皇帝被俘/受伤，朝纲崩盘
  └── ??? 意外：比如朱厚照在战场上遇到王阳明……
```

**史实 callback**：应州大捷是正德的 high point，LLM 推演可以复现或颠覆这段历史。

### 8.3 🎭 微服私访系统

**机制**：消耗 1 回合，皇帝偷跑出宫。

```
微服私访
  ├── → 随机触发民间事件
  │     ├── 市井奇遇（遇到人才/侠客/骗子）
  │     ├── 民情探查（发现某省真实民心，修正数值）
  │     ├── 官府黑幕（撞破贪腐，触发弹劾链）
  │     └── 突发危机（被认出身份/遇到刺客）
  │
  └── 后果：
        ├── 朝纲 -5（文官发现皇帝又跑了）
        ├── 威望 ±?（取决于微服结果）
        └── 玩心 +3
```

### 8.4 🗡 八虎博弈系统

正德初年最核心的朝堂博弈。

**八虎**：刘瑾、马永成、谷大用、张永等八位宦官，朱厚照做太子时的玩伴。

```
开局状态：
  ├── 刘瑾掌司礼监，权倾朝野
  ├── 八虎集体好感：80（开局极高）
  ├── 内阁/文官集团对八虎好感：-40
  └── 朝堂处于"皇帝被八虎包围"状态

博弈节点：
  ├── 杀刘瑾？
  │     ├── 需串通张永（八虎内部分化）
  │     ├── 成功后威望 +15，八虎好感 -30
  │     └── 失败后果：刘瑾反扑，朝纲 -30
  │
  ├── 保刘瑾？
  │     ├── 八虎好感 +10
  │     ├── 内阁好感 -20
  │     └── 贪腐持续上升
  │
  └── 平衡？
        ├── 两边都给甜头
        ├── 短期稳定但贪腐缓慢上升
        └── LLM 推演长期后果
```

**特色**：八虎不是简单的"奸臣"，他们是朱厚照在深宫里的玩伴和盟友。杀了刘瑾谁来陪你玩？这种情感纠葛是正德朝独有的推演空间。

### 8.5 📜 宁王/王阳明事件链

预置的关键历史事件线，但不是固定脚本——触发与否取决于玩家行为。

```
🔹 触发阶段（必要条件）：
  - 江西民心 < 50
  - 宗室关系 < 30
  - 朝纲 < 50
  → 宁王朱宸濠开始暗中筹备

🔹 发展阶段（LLM 推演）：
  - 宁王以"陛下被奸臣蒙蔽"为名养望
  - 王阳明此时在南赣巡抚任上
  - 玩家可以在以下节点干预：
    ① 提前调离宁王（需要确凿证据）
    ② 加强江西驻军
    ③ 启用王阳明督军
    ④ 或者无视警告……

🔹 爆发阶段：
  - 如果无人阻止 → 宁王正式造反
  - LLM 推演战局走向
  - 王阳明是否被启用 → 决定平叛难度
  - 玩家可选御驾亲征（复现历史）或派兵平叛

🔹 结局：
  - 平叛成功：王阳明封爵，心学崛起
  - 平叛失败：东南震动，可能亡国
  - 提前化解：改变历史走向
```

### 8.6 🎲 玩心系统（核心创新）

这是正德风云录跟所有历史模拟游戏最不同的地方。

**设计哲学**：
崇祯模拟器的核心驱动是"亡国焦虑"——玩家被推着走。
正德风云录的核心驱动是"玩 vs 治 的拉扯"——玩家自己跟自己较劲。

**机制**：

```
玩心(playfulness) 0–100
  │
  ├── 上升因素：
  │     ├── 去豹房: +5~10
  │     ├── 微服私访: +3
  │     ├── 听八虎怂恿: +2~5
  │     ├── 亲征（打爽了）: +8
  │     └── 长期不上朝: +2/回合
  │
  ├── 下降因素：
  │     ├── 勤政: -5~10/回合
  │     ├── 重大危机: -15（被吓到了）
  │     ├── 被文官死谏: -5
  │     └── 生病/受伤: -8
  │
  └── 不同区间效果：
        80-100 [狂浪] → 自动触发特殊剧情，玩家有时会被"拉去豹房"
        60-79  [贪玩] → 主动下诏倾向"任性"风格，决策理性下降
        40-59  [收敛] → 正常状态，玩家可以理性治国
        20-39  [沉稳] → 决策倾向保守，威望上升缓慢
        0-19   [消沉] → 精力下降，可能触发抑郁事件（朱厚照历史上也有低谷）
```

**玩法影响**：
- 玩心高了，LLM 在推演时会**倾向更激进/荒唐的后果**
- 玩心低了，LLM 推演会更加"老成持重"
- 玩心达到极值时，可能触发**专属剧情事件**（比如"陛下决定在豹房开科取士"）

## 11. Token 成本控制与付费模型

### 11.1 Token 消耗预估

| 场景 | 输入 Token | 输出 Token | 月估算（重度玩家） |
|------|-----------|-----------|------------------|
| 单次诏书推演 | 4,000–6,000 | 800–1,200 | — |
| 大臣对话 | 2,000–3,000 | 400–800 | — |
| 事件生成 | 1,500–2,500 | 300–500 | — |
| **单回合合计** | **5,500–8,500** | **1,100–1,700** | — |
| 100 回合/月 | — | — | 约 0.66–1.02 元 |
| 500 回合/月 | — | — | 约 3.3–5.1 元 |
| 1000 回合/月 | — | — | 约 6.6–10.2 元 |

> 基于 DeepSeek V3 价格（输入 ¥0.5/M，输出 ¥2/M）

### 11.2 优化策略

1. **Prompt 压缩**：世界状态用紧凑格式传递（`treasury:3000,granary:2000`），省 Token
2. **增量 Prompt**：状态变化部分只传 delta，不传全量
3. **记忆分级**：长期记忆只传关键摘要，不传全部历史
4. **缓存重复**：相同请求不重复调用（如同一条诏书点两次）
5. **模型分级**：日常推演用 DeepSeek（便宜），关键剧情用更强模型

### 11.3 付费模型建议

跟崇祯模拟器一致——**纯用户自接 API**：

```
App 免费
  ↓
用户自备 API Key（DeepSeek/千问/OpenRouter）
  ↓
App 直连 API，Token 消耗由用户直接付费给模型厂商
  ↓
我们零抽成、零服务费
```

**可选增值**（未来考虑）：
- Token 统计看板（免费）
- 存档云同步（可考虑收费）
- 官方推荐模型通道（加一点服务费，帮用户解决 API 充值难题）

## 13. 附录：与崇祯模拟器架构对比

| 维度 | 历史模拟器：崇祯 | 正德风云录（我们） |
|------|----------------|------------------|
| **底层架构** | 自研步进式状态机 + LLM | 同款架构，Android 原生实现 |
| **状态管理** | Python 服务端 + 关系型数据库 | Kotlin + Room，纯本地 |
| **LLM 调用** | 通过 OpenRouter（统一计费+代理） | OkHttp 直连，用户自配 API |
| **Prompt 工程** | 三层约束（状态/意图/持久化） | 同款三层 + 玩心参数注入 |
| **记忆系统** | 步进式增量 + 记忆压缩（放弃 RAG） | 同款 + 分层注入（Level 0-3） |
| **数值校验** | 后端硬约束 + LLM 软区间 | 同款，本地实现硬约束 |
| **特色系统** | 无（专注于核心推演） | 豹房/亲征/微服/八虎/心学事件链 |
| **部署** | Steam（PC） | Android（移动端） |
| **付费** | 买断+Token 计费（官方抽成） | 完全免费 + 玩家自接 API |
| **技术栈** | 未公开（推测 Python + Web） | Kotlin + Compose + Room |
| **团队** | 不到 10 人，制作人+全技术 | 1 人（我）+ 老板验收 😏 |
